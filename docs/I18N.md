# FAMII – Mehrsprachigkeit (i18n)

Zwischenphase: i18n **sauber vorbereitet**, Deutsch + Englisch aktiv. Keine
großen UI-Änderungen, keine neuen Features. Demo bleibt unverändert lauffähig.

## 1. Verwendetes i18n-System

Ein **leichtgewichtiges, eigenes System ohne externe Abhängigkeit** (kein
react-i18next/lingui nötig → kein nativer Rebuild, kein Risiko):

- Context-Provider `I18nProvider` mit `useI18n()` → `{ locale, setLocale, t }`.
- `t('namespace.key', { var })` mit Variablen-Interpolation (`{name}`).
- Wörterbücher als einfache verschachtelte Objekte.

## 2. Wo die Übersetzungen liegen

```
src/i18n/
├── translations.ts   ← Wörterbücher (de, en), SUPPORTED_LOCALES, PLANNED_LOCALES
├── I18nProvider.tsx  ← Provider, Geräteerkennung, Persistenz, t()/Fallback
└── index.ts          ← exportiert useI18n, I18nProvider, SUPPORTED_LOCALES …
```

- Eingebunden in `App.tsx` (Provider um die gesamte App).
- Gespeicherte Sprache: `AsyncStorage`-Schlüssel `famii.locale`.

## 3. Automatische Sprache & Auswahl

- **Erststart:** Geräte-/Browsersprache wird erkannt (`navigator.language`,
  Fallback `Intl`). Beginnt sie mit `de` → Deutsch, sonst Englisch.
- **Manuell:** Einstellungen → **„Sprache"** (`LanguageScreen`) → Deutsch /
  English. Auswahl wird gespeichert und sofort angewendet.

## 4. Neue Sprache ergänzen (z. B. Spanisch)

1. In `translations.ts` ein Wörterbuch `es` nach demselben Schema wie `de`/`en`
   anlegen und in `TRANSLATIONS` aufnehmen.
2. In `SUPPORTED_LOCALES` `{ code: 'es', label: 'Español' }` ergänzen und `es`
   aus `PLANNED_LOCALES` entfernen.
3. `Locale`-Typ um `'es'` erweitern.
Fertig – die Sprache erscheint automatisch in der Auswahl.

## 5. Fallback (kein leerer Text, kein Fehler)

`t()` sucht in dieser Reihenfolge:
**gewählte Sprache → Englisch → der Schlüssel selbst.**
Fehlt eine Übersetzung, erscheint also Englisch; im Notfall der Key-Text – nie
ein leerer String oder ein Absturz.

## 6. Geplante Sprachen (Struktur vorbereitet)

`PLANNED_LOCALES`: Español, Français, Italiano, Türkçe, العربية, Polski,
Português, Tagalog. (Anzeige „Bald verfügbar"; Übersetzungen folgen später.)

## 7. Design-Schutz (lange/kurze Texte)

- Migrierte Screens nutzen flexible Layouts (`flex`, `minWidth: 0`,
  `numberOfLines`, umbruchfähige Buttons) → keine abgeschnittenen Wörter,
  keine Überlappungen bei längeren EN/DE-Texten.
- Beim weiteren Auslagern bitte dieselbe Regel anwenden (Buttons/Labels nie auf
  feste Breite zwingen).

## 8. Status: was ist übersetzt / was ist offen

**Bereits über Translation-Keys (de + en):**
- Einstellungen (`SettingsScreen`) – Titel, Intro, alle Zeilen
- Sprachauswahl (`LanguageScreen`)
- gemeinsame Begriffe (`common.*`: Speichern, Abbrechen, Löschen, Fehler …)

**Struktur vorhanden, Inhalte folgen (noch hardcoded im UI):**
- Startseite, Onboarding, Familienbaum, Einladungen, Ehrenmitglieder,
  Erinnerungen, Dokumente, Zeitkapseln, Profil, Premium/Speicherhinweis,
  Fehlermeldungen.

Diese Bereiche funktionieren weiterhin vollständig auf Deutsch (kein
Regressionsrisiko); sie werden Schritt für Schritt auf Keys umgestellt, indem
ihre Texte nach `translations.ts` (Namespace je Bereich) wandern und im Screen
durch `t('…')` ersetzt werden – nach demselben Muster wie bei `SettingsScreen`.

## Sicherheit / Stabilität

- Kein neuer nativer Code, keine neue Runtime-Abhängigkeit.
- `tsc` fehlerfrei; Web-Render 0 Console-Errors; Demo unverändert.
- Live-Verhalten unverändert (Sprache wirkt clientseitig, kein Auth/Backend).
